# 🌐 DNS Resolution (সহজ ব্যাখ্যা)

আপনি যখন browser-এ `google.com` টাইপ করেন, তখন computer-কে বের করতে হয় এই নামের আসল **IP address** (যেমন `142.250.190.46`)।

👉 কারণ:
Internet নাম বোঝে না, শুধু **IP address** বোঝে।

---

# 🔄 পুরো Process ধাপে ধাপে

## 🟢 ধাপ ১ — Browser Cache

* Browser প্রথমে নিজের cache-এ খোঁজে
* আগে যদি IP resolve করা থাকে এবং TTL শেষ না হয়
  👉 তাহলে এখানেই IP পেয়ে যায় (no further lookup)

---

## 🟡 ধাপ ২ — OS (Operating System)

Browser না পেলে OS-কে জিজ্ঞেস করে:

1. `hosts` file চেক করে (manual mapping)
2. OS cache চেক করে

👉 যদি পাওয়া যায় → এখানেই শেষ

---

## 🔵 ধাপ ৩ — DNS Resolver

OS-ও না পেলে:

* Resolver-এ query পাঠায়
* Resolver হতে পারে:

  * Router
  * ISP (BTCL / GP)
  * Public DNS:

    * Google → `8.8.8.8`
    * Cloudflare → `1.1.1.1`

👉 Resolver প্রথমে নিজের cache চেক করে

---

## 🔴 ধাপ ৪ — Root Name Server

Resolver না পেলে:

* Root server-এ যায়
* পৃথিবীতে ১৩টি root server cluster আছে

👉 Root server শুধু বলে:

> “.com এর জন্য ওই server-এ যাও”

---

## 🟠 ধাপ ৫ — TLD Name Server

* `.com` → Verisign manage করে
* এখানে জিজ্ঞেস করা হয়:

👉 “google.com কোথায়?”

👉 TLD বলে:

> “এই Authoritative server-এ যাও”

---

## 🟣 ধাপ ৬ — Authoritative Name Server

* আসল উত্তর এখানেই থাকে
* Domain owner (Google) এটা control করে

👉 এখান থেকে পাওয়া যায়:
✔️ Final IP Address

---

## 🔁 ধাপ ৭ — IP ফিরে আসে

IP ফিরে আসে এই পথে:

```
Authoritative → Resolver → OS → Browser
```

👉 সবাই cache-এ রেখে দেয় (future speed-এর জন্য)

---

## 🔗 ধাপ ৮ — Actual Connection

IP পাওয়ার পর:

1. TCP connection
2. HTTPS handshake
3. Web page load

---

# 📌 সহজ উদাহরণ (Real-life analogy)

👉 এটা অনেকটা **ফোন ডিরেক্টরি**র মতো:

* আপনি নাম জানেন (google.com)
* কিন্তু নম্বর জানেন না (IP)

তাই:

1. নিজের কাছে খুঁজলেন
2. না পেলে লোকাল ডিরেক্টরি
3. না পেলে বড় অফিস
4. শেষে আসল নম্বর পেলেন
